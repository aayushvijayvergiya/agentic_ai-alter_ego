## Fix 1 â€” Gradio Version Mismatch (HF Build Failure)

**File**: `README.md`  
**Type**: Deployment / Build

### Root Cause
HF Spaces reads the `sdk_version` metadata from `README.md` and runs:
```
pip install -r requirements.txt gradio[oauth,mcp]=={sdk_version} ...
```
`README.md` had `sdk_version: 5.46.0`, but `requirements.txt` (generated by `uv export`) pinned `gradio==5.46.1`. Pip could not satisfy `==5.46.0` AND `==5.46.1` simultaneously, causing the build to fail with exit code 1.

### Fix
```diff
- sdk_version: 5.46.0
+ sdk_version: 5.46.1
```

### Lesson Learned
After running `uv sync` or `uv lock --upgrade`, always regenerate `requirements.txt` and verify `sdk_version` in `README.md` matches the exact `gradio` version pinned in `requirements.txt`. They must always be identical.

## Fix 2 â€” validate_config() Treating Optional Vars as Required (App Startup Failure)

**File**: `src/alter_ego/config.py`  
**Type**: Bug / Logic

### Root Cause
`validate_config()` listed `PUSHOVER_USER` and `PUSHOVER_TOKEN` in the required configs list. These are optional (notification-only). When these env vars are not set in HF Spaces (only `OPENAI_API_KEY` is configured there), `validate_config()` returned `False`, and `AlterEgoApp.__init__()` raised `ValueError("Invalid configuration")`, preventing the app from starting at all.

Additionally, `LINKEDIN_PDF_PATH` and `SUMMARY_FILE_PATH` were checked as `Path` objects â€” Python `Path` objects are always truthy regardless of whether the file exists, making those checks silently no-ops. `DocumentService` already handles missing files gracefully with fallback content.

### Fix
Simplified `validate_config()` to only require `OPENAI_API_KEY`:
```python
def validate_config(self) -> bool:
    if not self.openai_api_key:
        print("Missing required configuration: OPENAI_API_KEY")
        return False
    return True
```

### Lesson Learned
Only put truly blocking requirements in `validate_config()`. Optional integrations (Pushover, portfolio scraping) should be handled gracefully at the service level, not at startup validation.

## Fix 4 â€” Pytest Not Discoverable / Missing pythonpath Config

**File**: `pyproject.toml`  
**Type**: Developer Experience / Testing

### Root Cause
`uv run pytest tests/` failed with `ModuleNotFoundError: No module named 'src'` because pytest was not configured to add the project root to `sys.path`. Tests import `from src.alter_ego.config import Config`, which requires `.` to be on the Python path.

Additionally, `pytest` and `pytest-cov` were listed in `pyproject.toml` optional dev dependencies but had not been added via `uv add --dev`, so they were not installed in the venv.

### Fix
Added `[tool.pytest.ini_options]` to `pyproject.toml`:
```toml
[tool.pytest.ini_options]
pythonpath = ["."]
testpaths = ["tests"]
```
Installed dev deps:
```bash
uv add --dev pytest pytest-cov
```
